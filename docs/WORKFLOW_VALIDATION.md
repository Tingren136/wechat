# Workflow Validation

这份文档用于验证“公众号长文半自动化工作流”在封装成 skill 之前是否稳定。

## 当前要验证的主链路

1. 输入草稿正文
2. 长文润色
3. Markdown 结构化
4. 创建文章同名资产目录
5. 正文配图位分析
6. 图片提示词打包
7. 图片数量与方案确认
8. 正文图片生成
9. 图片插回正文
10. 阅读节奏检查
11. 输出 4 套 Wechat HTML
12. 用户手动选择排版
13. 导出选定主题的发布态 HTML
14. 进入草稿箱阶段

## 当前已确认可用

- 4 套 Wechat 排版原型可生成
- 同一篇文章可并行比较 4 套排版
- 正文图片图注、表格、宽度对比已经可视化
- 微信公众号草稿箱发稿链路曾经跑通
- 仓库内 `prototype/wechat-theme-previews/generate-previews.mjs` 已可直接重跑
- 4 套主题的发布态 HTML 已可导出
- 发布态已确认要用短图注与圆角显示层裁切

## 当前固定规则

- 正文图默认 `4:3` 横版
- 正文默认固定输出 4 套主题，不再收敛为单模板
- 配图、prompt、HTML 资产必须落在文章同名目录
- 连续纯文字区段不应超过约 `300` 中文字
- 预览页 HTML 不能直接用于公众号发稿
- 调用子 skill 时不能跳过其原始确认和中间产物
- 结尾不再插入签名图，也不再追加单独作者行

## 仍需重点验证

### A. 输入契约

- 输入正文是否必须带 frontmatter
- 图片目录是否必须固定命名为 `imgs`
- 图片提示词文件与图片文件是否必须同名
- 文章标题是否作为资产目录名唯一来源

### B. 中间产物契约

- 润色后的 Markdown 命名规则
- 配图建议文件结构
- prompt 打包结果结构
- 最终 HTML 输出结构
- 4 套发布态 HTML 的命名规则
- 文章目录内预览与发布态文件是否都必须保留

### C. 人工确认点

必须保留确认的阶段：

1. 润色后确认
2. 图片数量确认
3. 插图规划后确认
4. 图片生成前确认
5. 4 套排版中最终选择确认
6. 草稿箱投递前确认

### D. 阅读节奏

- 是否能自动识别长段纯文字区段
- 是否能在必要时补充视觉中断建议
- 是否能验证“约 300 中文字内至少一次视觉中断”

### E. 路径依赖

当前原型仍然依赖本地绝对路径，需要后续改成可配置路径。

## 下一步验证顺序

1. 先把“输入文章 -> 输出 4 套预览 + 4 套发布态 HTML”抽成最小稳定闭环
2. 再补“插图规划 + prompt 打包 + 阅读节奏检查”
3. 最后再接入整条完整工作流

## 判定通过标准

满足以下条件，才适合正式封装成 skill：

- 给定一篇标准 Markdown 正文，可以稳定产出 4 套 HTML
- 给定一篇标准 Markdown 正文，可以稳定产出 4 套发布态 HTML
- 所有中间产物命名和目录结构可预测
- 所有产物都能稳定落在文章同名目录中
- 每个阶段都能明确停下来让用户确认
- 视觉中断规则可以被解释和复核
- 新 AI 会话可以只读文档就继续推进
